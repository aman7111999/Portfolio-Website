<Section id="problem" eyebrow="01 · Problem" title="Every feature was paying an interest tax.">

Razorpay's merchant products had grown organically. Three parallel component libraries, four spacing scales, and every team maintaining its own button. New features were shipping on top of drift — the interest tax showed up as pixel-hunting reviews, subtle a11y regressions, and merchants complaining that "Razorpay doesn't feel like Razorpay anywhere."

<Callout kind="insight" title="The invisible cost">
Designers were spending roughly 30% of every sprint re-solving problems the system should have already answered. Engineers reimplemented the same primitives with subtle drift. The cost was invisible on any single feature — obvious across the roadmap.
</Callout>

</Section>

<Section id="research" eyebrow="02 · Research" title="Adoption is a UX problem, not a compliance problem.">

I audited 218 screens across six surfaces, tagged every component, and mapped drift. Interviewed 14 designers and 9 engineers about what a system would need to become the default choice.

</Section>

<PullQuote author="Priya Rao" role="Director of Design, Razorpay">
If the system is harder to adopt than to bypass, you don't have a system — you have a suggestion.
</PullQuote>

<Section id="principles" eyebrow="03 · Principles" title="Four commitments we'd hold each other to.">

<Callout kind="decision" title="Design principles">
1. **Composable, not comprehensive.** Ship primitives that combine, not screens that pretend.
2. **Defaults over documentation.** The right choice should be the easiest choice.
3. **Accessibility is a contract.** Every component ships with the a11y it promises, tested.
4. **Change is a feature.** Deprecation and migration are first-class product work.
</Callout>

</Section>

<Section id="approach" eyebrow="04 · Approach" title="A three-tier token architecture and a shared release rhythm.">

We rebuilt tokens as a three-tier hierarchy so teams could compose without forking. Components shipped with usage guidelines, an accessibility contract, and a Figma-to-code parity test. A weekly office hour turned complaints into upstream fixes.

<Section id="impact" eyebrow="06 · Impact" title="Six surfaces, ten months, one shared surface area.">

Six surfaces migrated in ten months. Design QA time dropped nearly 40%. The system now ships a release every two weeks with contributions from four different teams. More importantly, Blitz stopped being "the system team's project" — it became how we work.

<Callout kind="learning" title="What I'd do differently">
Start the contribution model in month one, not month eight. The hardest work wasn't the tokens or components; it was earning trust one PR review at a time until people stopped asking permission to use the system and started asking permission to extend it.
</Callout>

</Section>

<Section id="reflection" eyebrow="07 · Reflection" title="Systems are earned, not shipped.">

A design system succeeds when it dissolves into the work — when nobody remembers there was ever another way. That took a hundred small acts of maintenance, not a keynote reveal.

</Section>
