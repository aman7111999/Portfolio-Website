<Section id="problem" eyebrow="01 · Problem" title="Fluent, confident, and wrong.">

Meera was CRED's first user-facing AI product. It had to answer real money questions — "why did my payment fail?" — with humility, precision, and an obvious way out when it was wrong. Early prototypes were fluent but overconfident. Users treated Meera's guesses as facts, then blamed the product when a guess was wrong. Trust, once broken, was expensive to rebuild.

<Callout kind="warning" title="The trust cliff">
In diary studies, one wrong answer cost more trust than three correct answers earned. Confidence without calibration was worse than no answer at all.
</Callout>

</Section>

<Section id="research" eyebrow="02 · Research" title="Confidence is a visual language.">

Diary studies with 22 users showed that hedging phrases, source citations, and explicit "I don't know" options changed how people used the tool. Confidence isn't a number — it's a pattern the interface teaches.

</Section>

<PullQuote author="Diary study, participant 14">
I stopped double-checking Meera when it stopped acting like it knew everything.
</PullQuote>

<Section id="approach" eyebrow="03 · Approach" title="Three response states, one honest contract.">

<Steps items={[
  { title: "Answer with source",    body: "High-confidence response, cited. The primary path — used when the model has grounded data." },
  { title: "Best guess with verify", body: "Medium-confidence, hedged, with a one-tap 'verify' CTA that opens the source and lets the user confirm." },
  { title: "Graceful handoff",       body: "Low-confidence or out-of-scope: acknowledge the limit, route to a human, and preserve context." }
]} />

Every state ships with its own micro-interaction — the way a suggestion settles, the way a correction is acknowledged, the way a handoff explains itself.

</Section>

<Section id="system" eyebrow="04 · Trust system" title="Making uncertainty legible.">

<Callout kind="decision" title="Design decisions">
- **No confidence numbers.** Users don't parse "72%." They parse tone.
- **Sources are one tap away, always.** Verifying should never feel like doubting the product.
- **Corrections are cheap.** Every response has a "this is wrong" affordance that the model actually learns from.
</Callout>

<Divider label="Impact" />

</Section>

<Section id="impact" eyebrow="05 · Impact" title="Trust compounds when honesty is designed in.">

Task success rose 27% and support escalations dropped by nearly a third, even as usage grew. Users started forgiving mistakes because the product had already told them mistakes were possible.

</Section>

<Section id="reflection" eyebrow="06 · Reflection" title="Design the disclaimer, not just the answer.">

AI design is less about surface polish and more about the honesty of the interaction. What the model says matters less than how the product carries what it doesn't know.

</Section>
